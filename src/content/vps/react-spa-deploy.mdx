## When to use this variant

Choose this variant when your frontend is a SPA (Vite/CRA) and you need to pick where the build happens.

## Build strategy decision

- `ci-artifact` (recommended): build in GitHub Actions, deploy only generated files.
- `build-on-vps`: pull repo and run `npm ci && npm run build` directly on VPS.

## React-specific notes

- Ensure client-side routing fallback (`rewrite @spa /index.html` in Caddy).
- Serve only built output (`dist` for Vite, `build` for CRA).
- Keep hashed assets under `/assets/*` with long cache headers.

## CI artifact strategy (recommended)

~~~yaml
- name: Install deps and build
  run: |
    npm ci
    npm run build

- name: Upload dist to release dir
  run: |
    RR="/srv/www/<www-domain>/releases/${RELEASE_DIR}"
    ssh -o IdentitiesOnly=yes -i ~/.ssh/deploy_key "${{ secrets.VPS_USER }}@${{ secrets.VPS_HOST }}" "mkdir -p \"$RR\""
    rsync -az --delete -e "ssh -o IdentitiesOnly=yes -i ~/.ssh/deploy_key" dist/ "${{ secrets.VPS_USER }}@${{ secrets.VPS_HOST }}:${RR}/"
~~~

## Build on VPS strategy

Use only when CI build is not possible.

~~~bash
cd /srv/apps/<app-name>
git fetch --all --prune
git reset --hard origin/main
npm ci --no-audit --no-fund || npm install
npm run build
~~~

Then sync build output to the target release and switch `current`.

## Unified script usage for React mode

~~~bash
# Recommended
sudo bash scripts/deploy-vps-unified.sh \
  --app-type react \
  --server caddy \
  --strategy ci-artifact \
  --app-name <app-name> \
  --domain <domain> \
  --www-domain <www-domain> \
  --deploy-user <deploy-user>

# Alternative
sudo bash scripts/deploy-vps-unified.sh \
  --app-type react \
  --server caddy \
  --strategy build-on-vps \
  --app-name <app-name> \
  --domain <domain> \
  --www-domain <www-domain> \
  --repo https://github.com/<org>/<repo>.git \
  --deploy-user <deploy-user>
~~~
